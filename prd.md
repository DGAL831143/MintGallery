# MintGallery（薄荷绿家庭相册）产品需求文档

| 项目名称 | MintGallery | 文档版本 | V1.3.1（移动端断点续传版） |
| :--- | :--- | :--- | :--- |
| 创建时间 | 2026-06-17 | 修订时间 | 2026-06-22 |
| 当前状态 | v0.3.1 已实现 | 目标用户 | 个人及少量家庭成员 |
| 初始数据规模 | 约 100 GB，包含照片、普通视频和 Live Photo | 首期部署 | Windows 家用电脑、Tailscale 私有访问 |

> 重要说明：GitHub 仅用于保存项目代码，不用于存放照片、视频、数据库、密码或其他私人数据。首期媒体文件保存在运行后端服务的家用电脑硬盘中。

## 1. 项目背景与目标

### 1.1 核心问题

手机照片和视频占用大量存储空间，希望通过浏览器批量上传到自有设备，并以清晰、直观的图片墙方式浏览家庭影像。项目尤其关注苹果 HEIC 图片与 Live Photo 的保存和展示。

### 1.2 MVP 目标

首期完成以下闭环：

1. 管理员创建家庭账号。
2. 家庭成员通过手机或电脑浏览器登录。
3. 批量上传普通照片、HEIC 图片、普通视频和手动配对的 Live Photo。
4. 原始文件完整保存，后台生成适合网页浏览的缩略图或预览图。
5. 在绿色、简洁的响应式图片墙中浏览共享和个人内容。
6. 清楚展示文件处理状态和数据副本风险，避免误导用户删除唯一原件。
7. 已授权家庭设备通过 Tailscale 私有网络访问普通图片和视频上传、浏览功能。

### 1.3 MVP 不包含

- 面向公开互联网的匿名访问、公开分享链接和无需加入 tailnet 的远程访问。
- 使用 GitHub、GitHub Pages 或 Git 仓库存储媒体文件。
- 自动同步阿里云盘等商业网盘。
- 从 iPhone 相册自动提取并打包 Live Photo 双文件。
- 普通视频转码、视频压缩和多清晰度播放。
- 人脸识别、智能标签、地图、复杂相册分类和回收站。
- “上传后即可放心删除手机原件”的承诺。

## 2. 已确认的产品决策

| 决策项 | MVP 方案 | 后续方向 |
| :--- | :--- | :--- |
| 代码托管 | GitHub 私有仓库优先 | 持续集成与自动部署 |
| 媒体存储 | 家用电脑本地硬盘 | 网盘 API、对象存储或云硬盘 |
| 访问范围 | Tailscale 私有 tailnet，Fastify 同源提供页面和 API | 经安全评审后再考虑公开 HTTPS 访问 |
| 数据库 | SQLite | 并发和规模确有需要时再评估迁移 |
| 普通视频 | 支持上传和浏览器原生播放，不转码 | 异步转码和封面生成 |
| Live Photo | 保留手动双文件上传，暂缓体验优化 | 具备 macOS/Xcode 测试条件后再评估 iOS 上传器 |
| 第二副本 | MVP 不强制配置 | 外接硬盘或云端自动备份 |
| 管理方式 | 提供简单管理员页面 | 容量统计、日志和备份管理 |

## 3. 用户角色与核心场景

### 3.1 系统管理员

- 首次启动时创建唯一管理员账号。
- 创建、停用和重置家庭成员账号。
- 查看硬盘总容量、可用容量和媒体占用量。
- 查看失败的上传或处理任务并重试。
- 后续配置外接硬盘或云端备份。

### 3.2 家庭成员

- 使用管理员提供的账号登录，不开放公开注册。
- 批量选择照片和视频并查看逐项上传进度。
- 选择上传到“家庭共享”或“仅自己可见”。
- 浏览图片墙、查看大图、播放普通视频和 Live Photo。
- 明确知道文件是否处理完成，以及当前是否只有一个数据副本。

## 4. 信息架构与界面规范

### 4.1 页面结构

1. **登录页：** 账号、密码、登录按钮和简短错误提示。
2. **照片墙：** 登录后的默认页面，不设置营销首页。
3. **上传队列：** 展示总进度、单文件状态、失败原因、重试和取消操作。
4. **媒体查看器：** 全屏查看图片、普通视频或 Live Photo。
5. **管理员页面：** 账号管理、存储容量和任务状态。

### 4.2 视觉风格

- 风格关键词：简洁、直观、安静、以影像为主体。
- 主色：森林绿 `#2D6A4F`；强调色：薄荷绿 `#52B788`。
- 配合白色、浅灰和深灰文字，避免整个界面只有绿色。
- 页面不堆叠装饰卡片，不使用大面积渐变、光斑或营销式首屏。
- 操作按钮优先使用清晰图标；上传、删除、退出等重要操作同时提供文字或工具提示。
- 移动端主要操作区域满足触控尺寸要求，文字和按钮不得重叠或被截断。
- 颜色不能作为状态的唯一表达，错误、成功和处理中状态同时使用图标或文字。

### 4.3 图片墙

- 使用等高网格或稳定的瀑布流布局，兼容竖图、横图和全景图。
- 默认按拍摄时间倒序排列；缺少拍摄时间时按上传时间排序。
- 使用游标分页，每次加载固定数量，不一次请求全部记录。
- 首屏只请求轻量 WebP 缩略图，滚动接近视口时再懒加载后续内容。
- Live Photo 显示 `LIVE` 标识，普通视频显示播放图标和时长。
- 提供“家庭共享”和“仅自己可见”两个清晰视图。
- 包含加载中、空相册、处理失败、网络失败和重新加载状态。

### 4.4 全屏查看器

- 图片按屏幕尺寸加载预览图，不默认下载体积较大的原始文件。
- 支持左右滑动切换、双击放大和双指缩放。
- 提供查看信息和下载原始文件的明确入口。
- 普通视频使用浏览器原生播放器，并支持 HTTP Range 分段读取。
- 当前浏览器无法播放某种 MOV 编码时，显示“不支持在线播放”，但允许下载原文件。
- Live Photo 在用户长按或点击播放后才加载关联视频，松开或结束后回到静态图。

## 5. 功能需求

### 5.1 账号与权限

**[FR-101] 私密运行**

- 不提供公开注册、公开相册或匿名访问。
- 首次运行通过一次性初始化流程创建管理员，完成后关闭初始化入口。

**[FR-102] 登录保持**

- 使用服务端会话和 `HttpOnly`、`SameSite` Cookie 保持登录，不把长期令牌保存在网页可读取的本地存储中。
- 登录有效期建议为 30 天；管理员可使指定账号的现有会话失效。
- 本地开发可通过 `localhost` 使用；启用公网访问后必须使用 HTTPS 和 `Secure` Cookie。
- Tailscale 正式运行必须使用 Serve 提供的私有 HTTPS 地址；HTTP 端口只允许短时诊断。

**[FR-103] 可见范围**

- 每个资产拥有 `SHARED` 或 `PRIVATE` 可见范围。
- `SHARED` 对所有有效家庭成员可见。
- `PRIVATE` 仅上传者和管理员可见。
- 权限由数据库记录和后端接口判断，不能仅依赖磁盘目录名称。

**[FR-104] 管理员账号管理**

- 管理员可创建、停用账号和设置临时密码。
- 新用户首次登录后必须修改临时密码。
- 停用账号不会删除该账号上传的媒体。

### 5.2 批量上传

**[FR-201] 支持格式**

- 图片：JPEG、PNG、WebP、HEIC/HEIF。
- 普通视频：MP4、MOV。
- Live Photo：一张 HEIC/JPEG 静态文件和一个 MOV 视频文件。
- 文件是否被接受必须根据真实文件内容检测，不能只相信扩展名。

**[FR-202] 上传队列**

- 支持一次选择多个文件，建议至少能管理 50 个混合文件的队列。
- 显示总体进度和每个文件的等待、上传、校验、处理、完成或失败状态。
- 支持失败重试、取消未完成文件和重新选择文件。
- 浏览器进入后台、锁屏或被系统关闭时，上传可能暂停；页面恢复后继续未完成的分片。

**[FR-203] 分片与断点续传**

- 大文件采用成熟的 tus 或等价分片上传协议，不自行设计不兼容的私有协议。
- 服务端记录上传会话和已完成偏移量。
- 临时文件完成大小和 SHA-256 校验后，才能移动到正式原始文件目录。
- 服务重启后可以恢复有效的未完成会话，并定期清理过期临时文件。
- v0.3.1 普通照片和视频使用 `tus-js-client`、`@tus/server` 与本地 `FileStore`，按 1 MB 分片；连接中断时自动重试并查询服务端偏移后续传。
- tus 上传任务必须逐次验证 MintGallery 登录会话和任务所有者；完成后仍按真实文件内容检测格式，再写入正式原件目录与 SQLite。
- Live Photo 双文件入口暂不接入 tus，仍属于后续体验优化范围。

**[FR-204] 重复文件检测**

- 以 SHA-256、文件大小和资产类型辅助识别完全相同的文件。
- 检测到重复文件时提示用户跳过或保留副本，不未经确认自动删除。
- 哈希用于校验和查重，不能用于判断两个不同文件是否属于同一个 Live Photo。

### 5.3 原始文件和派生文件

**[FR-301] 保留原始文件**

- 上传的 HEIC、JPEG、MOV、MP4 等原始文件必须原样保存。
- 转码产物只能作为派生文件，不能覆盖或替代原始文件。
- 文件名由系统生成，原始文件名仅作为数据库元数据保存。

**[FR-302] 异步图片处理**

- 原始文件通过校验并落盘后，再进入后台处理队列。
- HEIC 解码环境必须显式验证 `libvips/libheif` 能力，不能假定 Sharp 默认安装一定支持所有 HEIC 文件。
- 处理任务使用有限并发，避免几十张高分辨率照片同时占满内存。
- 处理失败不影响原始文件保存，用户可稍后重试。

**[FR-303] 图片派生规格**

- 缩略图：长边不超过 480 px，WebP，去除不需要的元数据。
- 屏幕预览图：长边不超过 2048 px，WebP 或 JPEG，用于全屏浏览。
- 自动应用 EXIF 方向，防止图片旋转错误。
- 原始拍摄时间和必要 EXIF 信息单独写入数据库。

**[FR-304] 视频处理**

- MVP 原样保存普通视频，不做转码或压缩。
- 可读取时长、尺寸和拍摄时间，并生成视频封面；封面生成失败不阻止视频保存。
- 浏览器不支持的编码允许下载，不承诺所有 MOV 文件都能在线播放。

### 5.4 Live Photo

**[FR-401] 手动配对上传**

- MVP 不承诺网页直接从 iPhone 照片选择器获得完整 Live Photo 双文件。
- 用户需先把“未修改的原片”导出到“文件”，再在上传页选择对应的图片和 MOV。
- 上传页提供“作为 Live Photo 上传”入口和简短操作提示。
- v0.2.0 支持 `HEIC/HEIF/JPEG + MOV`，并要求两个文件的主文件名一致。
- 用户必须勾选确认两个文件来自同一张实况照片后才能上传。

**[FR-402] 配对识别**

- v0.2.0 使用相同主文件名和用户明确确认完成手动配对。
- 后续版本再读取图片和 MOV 内嵌的 Live Photo 资产标识进行自动校验。
- 系统不能确定时必须拒绝或要求用户确认，不能静默错误配对。

**[FR-403] 网页播放**

- 使用 Apple LivePhotosKit JS 或经过兼容验证的等价播放器。
- 图片墙仅显示轻量静态缩略图和 `LIVE` 标识，不预加载所有 MOV。
- 用户交互时才加载视频，加载失败时仍可查看静态图和下载原始双文件。
- v0.2.0 使用受保护的原始 MOV 地址，支持桌面悬停、手机长按和显式播放按钮。

### 5.5 状态与风险提示

每个资产至少具有以下处理状态：

| 状态 | 含义 | 用户提示 |
| :--- | :--- | :--- |
| `UPLOADING` | 文件仍在上传 | 请保持页面打开 |
| `VERIFYING` | 正在校验大小和哈希 | 正在验证文件完整性 |
| `PROCESSING` | 原始文件已保存，正在生成预览 | 暂时可能无法预览 |
| `READY` | 原始文件和必要预览已就绪 | 上传完成，但当前可能只有一个副本 |
| `FAILED` | 上传或处理失败 | 展示原因和重试操作 |

在未配置第二副本时：

- 所有已完成资产显示“仅 1 个副本”的风险状态。
- 完成提示不得出现“可以放心删除手机原件”等表述。
- 应提示用户暂时保留手机原件，或自行复制到另一块物理硬盘。

配置第二副本后，可增加 `BACKED_UP` 状态。只有原始文件与数据库记录均完成第二副本并通过校验后，才能显示“已安全备份”。

## 6. 数据与存储设计

### 6.1 本地目录

生产数据根目录必须通过配置指向 Git 仓库之外的独立磁盘目录，内部继续使用相对路径组织文件：

```text
F:/MintGallery/data/
  originals/{userId}/{year}/{month}/{assetId}/
  derivatives/{assetId}/
  temporary/{uploadId}/
  database/
```

- 数据库只保存相对路径，不保存依赖某台电脑盘符的绝对路径。
- 启动脚本必须拒绝位于项目仓库内部的数据根目录；最终盘符由部署时显式确认。
- 原始文件、派生文件和临时文件分开存放。
- 共享或私密权限不通过物理目录直接暴露。
- 磁盘剩余空间不足时拒绝开始新上传，并保留明确错误信息。

### 6.2 核心数据表

**`users`**

| 字段 | 说明 |
| :--- | :--- |
| `id` | 用户 UUID |
| `username` | 唯一登录名 |
| `password_hash` | 密码哈希 |
| `role` | `ADMIN` 或 `MEMBER` |
| `status` | `ACTIVE` 或 `DISABLED` |
| `created_at` | 创建时间 |

**`assets`**

| 字段 | 说明 |
| :--- | :--- |
| `id` | 资产 UUID |
| `owner_id` | 上传者用户 ID |
| `type` | `IMAGE`、`VIDEO` 或 `LIVE_PHOTO` |
| `visibility` | `SHARED` 或 `PRIVATE` |
| `status` | 上传与处理状态 |
| `shooting_time` | 原始拍摄时间，可空 |
| `uploaded_at` | 上传完成时间 |
| `width`、`height` | 图片或视频尺寸，可空 |
| `duration_ms` | 视频时长，可空 |
| `backup_status` | `NOT_CONFIGURED`、`PENDING`、`BACKED_UP` 或 `FAILED` |

**`asset_files`**

| 字段 | 说明 |
| :--- | :--- |
| `id` | 文件 UUID |
| `asset_id` | 所属资产 ID |
| `kind` | `ORIGINAL_IMAGE`、`ORIGINAL_VIDEO`、`THUMBNAIL`、`PREVIEW` 或 `POSTER` |
| `relative_path` | 相对于数据根目录的路径 |
| `original_name` | 用户上传时的原始文件名 |
| `mime_type` | 经服务端检测的 MIME 类型 |
| `size_bytes` | 文件大小 |
| `sha256` | 完整性校验值 |

还需使用 `upload_sessions` 记录分片上传进度，使用 `processing_jobs` 记录可重试的派生文件任务。

## 7. 安全要求

**[NFR-101] 媒体访问控制**

- 浏览器通过安全会话 Cookie 访问受保护媒体，或使用短时有效的签名 URL。
- 后端必须逐次检查资产可见范围，不能把媒体目录作为公开静态目录。
- 不向前端返回服务器真实磁盘路径。
- 视频接口支持鉴权后的 Range 请求。

**[NFR-102] 账号安全**

- 密码使用 Argon2id 或等价的现代密码哈希算法保存。
- 登录接口具备速率限制和失败日志。
- Cookie 会话启用 CSRF 防护；公网阶段必须启用 HTTPS。

**[NFR-103] 上传安全**

- 限制单文件大小、单次队列数量和允许的真实文件类型。
- 清理文件名并阻止目录穿越。
- 异常、磁盘写满或处理中断时，不生成半完成的正式资产记录。

**[NFR-104] GitHub 安全**

- `.gitignore` 必须排除媒体目录、SQLite 数据库、环境变量文件、日志和临时文件。
- 提交前检查密码、密钥、会话密钥和私人文件。
- GitHub 仓库优先设为私有，但私有仓库也不能用于提交家庭照片。

**[NFR-105] Tailscale 私有访问**

- Fastify 只监听 `127.0.0.1`，不直接开放局域网或公网端口。
- Tailscale Serve 负责 tailnet 内的 HTTPS 终止和访问路径，不使用 Funnel。
- 只有管理员批准加入同一 tailnet 的设备才能访问相册地址。
- 服务端 Cookie 在 HTTPS 模式启用 `Secure`，账号验证仍由 MintGallery 自身完成。
- Windows 电脑必须配置开机启动、禁止自动休眠，并保留可检查的运行日志。

## 8. 数据安全与备份策略

### 8.1 MVP 现状

MVP 默认只有家用电脑硬盘上的一个副本。该方案可以验证上传和浏览功能，但不能作为长期可靠的照片归档方案。硬盘损坏、误删除、勒索软件或电脑丢失都可能造成永久数据丢失。

### 8.2 推荐的后续方案

- 第二副本必须位于另一块物理硬盘或云端，不能只是同一硬盘的另一个分区。
- 至少备份原始媒体文件、数据库和必要配置；缩略图可以重新生成。
- 备份任务记录时间、文件数量、校验结果和失败原因。
- 定期执行恢复测试，确认备份不是只有记录而无法恢复。
- 后续接入网盘时应通过独立存储适配层实现，避免修改相册和权限核心逻辑。

## 9. 性能与兼容性

### 9.1 MVP 性能目标

以下指标在家用局域网、首批 30 个媒体项目、使用缩略图的条件下验收：

- 照片墙首屏主要内容在 2 秒内可见。
- 单张缩略图建议控制在 100 KB 以内。
- 列表采用分页和懒加载，媒体总量增加时不会一次载入全部数据。
- 图片处理并发可配置，默认不超过 2 个任务，避免家用电脑内存骤增。
- 100 GB 已有媒体可以分批导入，单批失败不影响已经完成的文件。

### 9.2 兼容性目标

- 优先支持近期版本的 iOS Safari、Android Chrome 和桌面 Chrome/Edge。
- HEIC、MOV 和 Live Photo 必须使用真实 iPhone 样本测试，不能只使用改扩展名的模拟文件。
- 浏览器后台上传和视频编码兼容性属于平台限制，界面必须提供明确反馈和恢复方式。

公网 4G 下的首屏性能、弱网策略和跨地区访问在公网部署阶段另行定义，MVP 不承诺固定 1.5 秒指标。

## 10. MVP 验收标准

满足以下条件才视为本地存储 MVP 完成：

1. 管理员能够创建和停用家庭成员账号。
2. 未登录用户无法查看资产列表或通过猜测 URL 获取媒体文件。
3. 用户能够上传至少 50 个混合图片和视频文件，并看到逐项进度与失败重试。
4. 大文件上传中断后能够从已确认分片继续，而不是从零开始。
5. 上传后的原始文件 SHA-256 与客户端选择的文件一致，HEIC 原文件未被转码覆盖。
6. JPEG、PNG、WebP 和受支持的 HEIC 样本能够生成缩略图和预览图。
7. 普通 MP4/MOV 可以保存；浏览器支持的编码可以播放，不支持的编码可以下载。
8. 用户可以手动上传并确认一组 Live Photo 配对文件，网页能够播放或回退为静态图。
9. 共享资产对所有有效成员可见，私密资产不能被其他普通成员访问。
10. 磁盘空间不足、格式错误和处理失败均有可理解的错误提示，不留下可见的损坏资产。
11. 未配置第二副本时，界面始终显示“仅 1 个副本”，且不提示用户安全删除手机原件。
12. Git 仓库中不包含媒体、数据库、密码、密钥和临时上传文件。

## 11. 技术实现建议

以下是实现建议，不作为用户可见功能：

- 前端：Vue 3、TypeScript、Vite、Vue Router、Pinia。
- 后端：Node.js、TypeScript、Fastify 或等价成熟框架。
- 数据库：SQLite，启用 WAL 模式和迁移机制。
- 图片处理：Sharp，并在目标系统验证 HEIC/HEIF 解码依赖。
- 断点续传：tus 协议及成熟服务端实现。
- 媒体存储：通过统一存储接口封装本地文件系统，后续增加网盘或对象存储适配器。
- 后台任务：MVP 使用数据库记录的持久化任务队列，限制并发并支持服务重启后恢复。

## 12. 路线图

### Phase 1：本地存储 MVP

- 局域网登录和账号管理。
- 普通照片、HEIC、普通视频上传。
- 分片续传、原件保留、缩略图和预览图。
- 图片墙、全屏查看和权限隔离。
- 手动配对 Live Photo。
- 单副本风险提示。

### Phase 2：私有在线访问与可靠运行

- Tailscale Serve 私有 HTTPS、同源页面与 API。
- Windows 后台运行、开机启动、日志和健康检查。
- 普通图片与视频在移动网络条件下的上传和浏览验收。
- 外接硬盘或云端第二副本。
- 备份校验、恢复测试和“已安全备份”状态。
- 网盘 API、对象存储或云硬盘适配器。
- 普通视频异步转码。

### Phase 3：体验增强

- iOS 快捷指令或专用工具辅助上传 Live Photo 原始双文件。
- 时间轴筛选、自定义相册、回收站和批量管理。
- 智能标签、搜索和更完整的存储统计。

## 13. 已知限制

- 纯网页无法保证直接从 iPhone 照片选择器取得完整 Live Photo 双文件。
- 浏览器在锁屏或后台停留过久时可能暂停上传，断点续传只能在页面恢复后继续。
- 不转码时，部分 MOV 视频可能无法在非苹果浏览器中播放。
- 本地只有一个副本时，不能把该系统视为安全备份，也不建议删除手机上的唯一原件。
- GitHub Pages 只能托管静态前端，不能承担本项目的 Node.js 后端和本地媒体存储。
- Tailscale 模式要求访问设备安装客户端并加入同一 tailnet；电脑关机、休眠或断网时相册不可访问。
- Live Photo 自动提取和便捷上传暂缓，当前手动双文件入口仅作为备用能力保留。
