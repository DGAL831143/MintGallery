# MintGallery（薄荷绿家庭相册）产品需求文档

| 项目名称 | MintGallery | 文档版本 | V1.20（共用展示页版） |
| :--- | :--- | :--- | :--- |
| 创建时间 | 2026-06-17 | 修订时间 | 2026-07-02 |
| 当前状态 | v1.3.0 已新增共用 MintGallery 展示页 | 目标用户 | 个人及少量家庭成员 |
| 初始数据规模 | 约 100 GB，包含照片、普通视频和 Live Photo | 首期部署 | Windows 家用电脑、Tailscale 私有访问 |

> 重要说明：GitHub 仅用于保存项目代码，不用于存放照片、视频、数据库、密码或其他私人数据。首期媒体文件保存在运行后端服务的家用电脑硬盘中。

## 1. 项目背景与目标

### 1.1 核心问题

手机照片和视频占用大量存储空间，希望通过浏览器批量上传到自有设备，并以清晰、直观的图片墙方式浏览家庭影像。项目尤其关注苹果 HEIC 图片与 Live Photo 的保存和展示。

### 1.2 MVP 目标

首期完成以下闭环：

1. 管理员创建家庭账号。
2. 家庭成员通过手机或电脑浏览器登录。
3. 批量上传普通照片、HEIC 图片、普通视频和手动配对的 Live Photo。
4. 原始文件完整保存，后台生成适合网页浏览的缩略图或预览图。
5. 在绿色、简洁的响应式图片墙中浏览共享和个人内容。
6. 清楚展示文件处理状态和数据副本风险，避免误导用户删除唯一原件。
7. 已授权家庭设备通过 Tailscale 私有网络访问普通图片和视频上传、浏览功能。
8. 在非宿主设备上优先加载轻量派生文件，并为 Live Photo 提供兼容、可回退的远程播放体验。

### 1.3 MVP 不包含

- 面向公开互联网的匿名访问、公开分享链接和无需加入 tailnet 的远程访问。
- 使用 GitHub、GitHub Pages 或 Git 仓库存储媒体文件。
- 自动同步阿里云盘等商业网盘。
- 从 iPhone 相册自动提取并打包 Live Photo 双文件。
- 普通视频转码、视频压缩和多清晰度播放。
- 人脸识别、自动智能标签、地图和复杂相册分类。
- “上传后即可放心删除手机原件”的承诺。

## 2. 已确认的产品决策

| 决策项 | MVP 方案 | 后续方向 |
| :--- | :--- | :--- |
| 代码托管 | GitHub 私有仓库优先 | 持续集成与自动部署 |
| 媒体存储 | 家用电脑本地硬盘 | 网盘 API、对象存储或云硬盘 |
| 访问范围 | Tailscale 私有 tailnet，Fastify 同源提供页面和 API | 经安全评审后再考虑公开 HTTPS 访问 |
| 数据库 | SQLite | 并发和规模确有需要时再评估迁移 |
| 普通视频 | 支持上传和浏览器原生播放，不转码 | 异步转码和封面生成 |
| Live Photo | 保留手动双文件上传；v0.3.2 优化远程播放，不改变上传入口 | 具备 macOS/Xcode 测试条件后再评估 iOS 上传器 |
| 第二副本 | MVP 不强制配置 | 外接硬盘或云端自动备份 |
| 管理方式 | 提供简单管理员页面 | 容量统计、日志和备份管理 |

## 3. 用户角色与核心场景

### 3.1 系统管理员

- 首次启动时创建唯一管理员账号。
- 创建、停用和重置家庭成员账号。
- 查看硬盘总容量、可用容量和媒体占用量。
- 查看失败的上传或处理任务并重试。
- 后续配置外接硬盘或云端备份。

### 3.2 家庭成员

- 使用管理员提供的账号登录，不开放公开注册。
- 批量选择照片和视频并查看逐项上传进度。
- 选择上传到“家庭共享”或“仅自己可见”。
- 浏览图片墙、查看大图、播放普通视频和 Live Photo。
- 明确知道文件是否处理完成，以及当前是否只有一个数据副本。

## 4. 信息架构与界面规范

### 4.1 页面结构

1. **登录页：** 账号、密码、登录按钮和简短错误提示。
2. **精选：** 登录后的默认页面，不设置营销首页；使用真实缩略图轮转墙展示回忆入口，例如今日往年和本月回忆。
3. **共用展示页：** 点击顶部 `MintGallery` 文字或叶子图标进入沉浸式图片浏览界面；默认使用收藏的家庭共享照片，管理员可修改全局展示图片，所有用户共用同一套配置。
4. **上传队列：** 展示总进度、单文件状态、失败原因、重试和取消操作。
5. **媒体查看器：** 全屏查看图片、普通视频或 Live Photo。
6. **管理员页面：** 账号管理、存储容量和任务状态。
7. **个人文件夹：** 创建逻辑文件夹、批量归类照片并按文件夹浏览。
8. **时间轴：** 按照片拍摄月份分段浏览，并支持按月份筛选；桌面端提供右侧年份导航。
9. **全部照片：** 完整照片列表入口，提供密度切换、媒体类型筛选、整理状态筛选、选择模式和分页浏览。
10. **标签、搜索与信息面板：** 在当前权限和筛选范围内搜索照片，并查看/修改照片标签、基础信息、可见范围、防窥状态、收藏和最近删除状态。

### 4.2 视觉风格

- 风格关键词：简洁、直观、安静、以影像为主体。
- 主色：森林绿 `#2D6A4F`；强调色：薄荷绿 `#52B788`。
- 配合白色、浅灰和深灰文字，避免整个界面只有绿色。
- 页面不堆叠装饰卡片，不使用大面积渐变、光斑或营销式首屏。
- 操作按钮优先使用清晰图标；上传、删除、退出等重要操作同时提供文字或工具提示。
- 移动端主要操作区域满足触控尺寸要求，文字和按钮不得重叠或被截断。
- 颜色不能作为状态的唯一表达，错误、成功和处理中状态同时使用图标或文字。

### 4.3 图片墙

- 使用等高网格或稳定的瀑布流布局，兼容竖图、横图和全景图。
- 默认按拍摄时间倒序排列；缺少拍摄时间时按上传时间排序。
- 使用游标分页，每次加载固定数量，不一次请求全部记录。
- 首屏只请求轻量 WebP 缩略图，滚动接近视口时再懒加载后续内容。
- Live Photo 显示 `LIVE` 标识，普通视频显示播放图标和时长。
- 提供“家庭共享”和“仅自己可见”两个清晰视图。
- 包含加载中、空相册、处理失败、网络失败和重新加载状态。
- 支持进入选择模式后批量勾选媒体，并加入个人文件夹或从当前文件夹移出。
- 支持批量设置标签、收藏/取消收藏、设置或取消防窥遮挡、调整家庭共享/仅自己可见，以及移入最近删除。
- 防窥照片在精选、全部照片、时间轴、文件夹和搜索结果中必须显示模糊或马赛克遮挡，不直接暴露清晰缩略图。
- 收藏入口显示已收藏资产；最近删除入口显示已软删除资产，普通图库、时间轴、文件夹和搜索结果默认排除最近删除内容。
- 文件夹使用横向标签栏切换，不改变“家庭共享”和“仅自己可见”的权限筛选。
- 桌面端使用常驻图库侧栏切换精选、时间轴、全部照片、收藏、最近删除、文件夹和月份；移动端使用横向入口栏、精选/时间轴切换和月份选择器。
- 桌面端时间轴右侧提供真实时间比例 scrubber：年份按日历时间分布，有照片的月份显示为小点，悬停小点显示月份，点击小点或年份可快速筛选。
- 精选页使用真实缩略图轮转墙展示回忆类入口，例如今日往年和本月回忆；点击轮转照片或中间入口进入对应普通列表。
- 共用展示页使用独立全屏布局展示管理员选择的全局图片集；未设置时从收藏的家庭共享照片回退生成。展示页仅展示家庭共享、非防窥、已处理完成的图片和 Live Photo 静态图，最多 13 张，不复制或移动原始文件。
- 时间轴使用稳定方格并按月份分段，全部照片保留适应横图、竖图和全景图的比例布局。
- 全部照片和精选集列表提供大图、标准和紧凑三种密度；手机端分别对应单列、两列和三列浏览，桌面端按窗口宽度自适应。
- 快速筛选支持媒体类型（全部、照片、视频、实况）和整理状态（全部、最近导入、待整理、防窥）。
- “最近导入”按上传或外部导入时间倒序展示；“待整理”表示尚未设置标签的资产；二者属于全部照片中的整理筛选，不放入精选页。“今日往年”和“本月回忆”只根据真实拍摄时间计算，不使用上传时间冒充回忆。该入口不改变时间轴继续按拍摄时间展示的原则。

### 4.4 全屏查看器

- 图片按屏幕尺寸加载预览图，不默认下载体积较大的原始文件。
- 查看器默认必须完整显示当前图片或视频，不得因视口高度、图片宽高比或手机底部翻页按钮导致内容被裁切；竖图、横图、超宽图、超长图和编辑后的派生图都应以 `contain` 方式适配舞台区域。
- 查看器先显示静态预览；Live Photo 的动态文件只在用户点击或长按后请求。
- 支持左右滑动切换、双击放大和双指缩放。
- 提供查看信息和下载原始文件的明确入口。
- 查看器标题优先显示用户设置的标签；原始上传文件名只作为技术元数据保留，不作为主要展示名称。
- 查看器提供收藏入口；上传者或管理员可以从查看器将资产移入最近删除或从最近删除恢复。
- 防窥照片进入查看器后仍先显示遮挡层，用户必须再次点击“查看原图”才显示清晰预览或触发实况播放；关闭查看器或切换项目后重新遮挡。
- 普通视频使用浏览器原生播放器，并支持 HTTP Range 分段读取。
- 当前浏览器无法播放某种 MOV 编码时，显示“不支持在线播放”，但允许下载原文件。
- Live Photo 在用户长按或点击播放后才加载关联视频，松开或结束后回到静态图。
- 查看器为上传者和管理员提供“编辑照片”入口，支持普通图片和 Live Photo 静态图的裁剪、旋转、水平翻转和垂直翻转；编辑只替换当前网页缩略图/预览图，不修改原始文件，也不编辑 Live Photo 的 MOV 动态部分。

## 5. 功能需求

### 5.1 账号与权限

**[FR-101] 私密运行**

- 不提供公开注册、公开相册或匿名访问。
- 首次运行通过一次性初始化流程创建管理员，完成后关闭初始化入口。

**[FR-102] 登录保持**

- 使用服务端会话和 `HttpOnly`、`SameSite` Cookie 保持登录，不把长期令牌保存在网页可读取的本地存储中。
- 登录有效期建议为 30 天；管理员可使指定账号的现有会话失效。
- 本地开发可通过 `localhost` 使用；启用公网访问后必须使用 HTTPS 和 `Secure` Cookie。
- Tailscale 正式运行必须使用 Serve 提供的私有 HTTPS 地址；HTTP 端口只允许短时诊断。

**[FR-103] 可见范围**

- 每个资产拥有 `SHARED` 或 `PRIVATE` 可见范围。
- `SHARED` 对所有有效家庭成员可见。
- `PRIVATE` 仅上传者和管理员可见。
- 权限由数据库记录和后端接口判断，不能仅依赖磁盘目录名称。

**[FR-104] 管理员账号管理**

- 管理员可创建、停用账号和设置临时密码。
- 新用户首次登录后必须修改临时密码。
- 停用账号不会删除该账号上传的媒体。

### 5.2 批量上传

**[FR-201] 支持格式**

- 图片：JPEG、PNG、WebP、HEIC/HEIF。
- 普通视频：MP4、MOV。
- Live Photo：一张 HEIC/JPEG 静态文件和一个 MOV 视频文件。
- 文件是否被接受必须根据真实文件内容检测，不能只相信扩展名。

**[FR-202] 上传队列**

- 支持一次选择多个文件，建议至少能管理 50 个混合文件的队列。
- 显示总体进度和每个文件的等待、上传、校验、处理、完成或失败状态。
- 支持失败重试、取消未完成文件和重新选择文件。
- 浏览器进入后台、锁屏或被系统关闭时，上传可能暂停；页面恢复后继续未完成的分片。

**[FR-203] 分片与断点续传**

- 大文件采用成熟的 tus 或等价分片上传协议，不自行设计不兼容的私有协议。
- 服务端记录上传会话和已完成偏移量。
- 临时文件完成大小和 SHA-256 校验后，才能移动到正式原始文件目录。
- 服务重启后可以恢复有效的未完成会话，并定期清理过期临时文件。
- v0.3.1 普通照片和视频使用 `tus-js-client`、`@tus/server` 与本地 `FileStore`，按 1 MB 分片；连接中断时自动重试并查询服务端偏移后续传。
- tus 上传任务必须逐次验证 MintGallery 登录会话和任务所有者；完成后仍按真实文件内容检测格式，再写入正式原件目录与 SQLite。
- Live Photo 双文件入口暂不接入 tus，仍属于后续体验优化范围。

**[FR-204] 重复文件检测**

- 以 SHA-256、文件大小和资产类型辅助识别完全相同的文件。
- 检测到重复文件时提示用户跳过或保留副本，不未经确认自动删除。
- 哈希用于校验和查重，不能用于判断两个不同文件是否属于同一个 Live Photo。

**[FR-205] 外部文件夹扫描导入**

- 管理员可以输入宿主 Windows 电脑上的绝对文件夹路径并触发扫描。
- 扫描过程只读取源文件，不移动、不重命名、不删除源文件。
- 扫描路径不能是 MintGallery 当前数据目录，也不能包含当前数据目录。
- 服务端按真实文件内容识别 JPEG、PNG、WebP、HEIC/HEIF、MP4 和 MOV，不只信任扩展名。
- 扫描结果必须展示新项目、重复项、Live Photo 候选和跳过项，导入前由管理员确认。
- 导入时复制选中的文件到正式数据目录，再复用现有上传入库、缩略图、预览图和 Live Photo 派生文件流程。

### 5.3 原始文件和派生文件

**[FR-301] 保留原始文件**

- 上传的 HEIC、JPEG、MOV、MP4 等原始文件必须原样保存。
- 转码产物只能作为派生文件，不能覆盖或替代原始文件。
- 文件名由系统生成，原始文件名仅作为数据库元数据保存。

**[FR-302] 异步图片处理**

- 原始文件通过校验并落盘后，再进入后台处理队列。
- HEIC 解码环境必须显式验证 `libvips/libheif` 能力，不能假定 Sharp 默认安装一定支持所有 HEIC 文件。
- 处理任务使用有限并发，避免几十张高分辨率照片同时占满内存。
- 处理失败不影响原始文件保存，用户可稍后重试。

**[FR-303] 图片派生规格**

- 缩略图：长边不超过 480 px，WebP，去除不需要的元数据。
- 屏幕预览图：长边不超过 2048 px，WebP 或 JPEG，用于全屏浏览。
- 自动应用 EXIF 方向，防止图片旋转错误。
- 原始拍摄时间和必要 EXIF 信息单独写入数据库。

**[FR-303A] 非破坏式网页图片编辑**

- 上传者或管理员可以在查看器中编辑普通图片和 Live Photo 的静态图片。
- 支持裁剪、90 度步进旋转、水平翻转和垂直翻转。
- 原始文件必须保持不变；编辑结果仅作为新的 `EDIT_THUMBNAIL` 和 `EDIT_PREVIEW` 派生文件保存。
- 每个资产同一时间只有一个有效编辑版本；恢复原图会取消当前有效编辑版本，并重新使用原始缩略图和预览图。
- 编辑后的资产需要在缩略图、查看器底部状态和信息面板中显示“已编辑”状态。
- 当前版本不支持普通视频编辑、Live Photo 动态视频编辑、滤镜、涂鸦、文字、批量编辑或完整编辑历史 UI。

**[FR-304] 视频处理**

- MVP 原样保存普通视频，不做转码或压缩。
- 可读取时长、尺寸和拍摄时间，并生成视频封面；封面生成失败不阻止视频保存。
- 浏览器不支持的编码允许下载，不承诺所有 MOV 文件都能在线播放。
- v0.3.2 只为 Live Photo 生成网页播放派生文件，不在本版本扩展为普通视频转码系统。

### 5.4 Live Photo

**[FR-401] 手动配对上传**

- MVP 不承诺网页直接从 iPhone 照片选择器获得完整 Live Photo 双文件。
- 用户需先把“未修改的原片”导出到“文件”，再在上传页选择对应的图片和 MOV。
- 上传页提供“作为 Live Photo 上传”入口和简短操作提示。
- v0.2.0 支持 `HEIC/HEIF/JPEG + MOV`，并要求两个文件的主文件名一致。
- 用户必须勾选确认两个文件来自同一张实况照片后才能上传。

**[FR-402] 配对识别**

- v0.2.0 使用相同主文件名和用户明确确认完成手动配对。
- v0.6.0 在外部文件夹扫描中，将同一文件夹内唯一同名 `HEIC/HEIF/JPEG + MOV` 标记为 Live Photo 候选。
- 后续版本再读取图片和 MOV 内嵌的 Live Photo 资产标识进行自动校验。
- 系统不能确定时必须拒绝或要求用户确认，不能静默错误配对。

**[FR-403] 网页播放**

- 使用经目标浏览器验证的原生组合播放器或等价成熟播放器，不把 Apple 专用播放器作为唯一实现路径。
- 图片墙仅显示轻量静态缩略图和 `LIVE` 标识，不预加载所有 MOV。
- 用户交互时才加载视频，加载失败时仍可查看静态图和下载原始双文件。
- v0.2.0 使用受保护的原始 MOV 地址，支持桌面悬停、手机长按和显式播放按钮。
- v0.3.2 保留原始 MOV，同时生成最长边不超过 1280 px、H.264 视频、适合流式播放的 MP4 派生文件；存在音轨时转为 AAC，网页默认播放派生文件。
- MP4 必须支持鉴权后的 HTTP Range 请求，并将流式播放所需元数据放在文件前部，避免下载完整视频后才能起播。
- 派生文件尚未生成或转码失败时，静态照片仍须立即可看，并提供重试与原始文件入口。

### 5.5 状态与风险提示

每个资产至少具有以下处理状态：

| 状态 | 含义 | 用户提示 |
| :--- | :--- | :--- |
| `UPLOADING` | 文件仍在上传 | 请保持页面打开 |
| `VERIFYING` | 正在校验大小和哈希 | 正在验证文件完整性 |
| `PROCESSING` | 原始文件已保存，正在生成预览 | 暂时可能无法预览 |
| `READY` | 原始文件和必要预览已就绪 | 上传完成，但当前可能只有一个副本 |
| `FAILED` | 上传或处理失败 | 展示原因和重试操作 |

在未配置第二副本时：

- 所有已完成资产显示“仅 1 个副本”的风险状态。
- 完成提示不得出现“可以放心删除手机原件”等表述。
- 应提示用户暂时保留手机原件，或自行复制到另一块物理硬盘。

配置第二副本后，可增加 `BACKED_UP` 状态。只有原始文件与数据库记录均完成第二副本并通过校验后，才能显示“已安全备份”。

### 5.6 远程浏览与弱网反馈

**[FR-501] 分层加载**

- 精选、全部照片和其他普通列表只加载缩略图；打开查看器后加载屏幕预览；只有下载操作才请求图片原件。
- Live Photo 先显示静态预览，用户触发播放后才请求动态派生文件。
- 查看器只允许预取前后各一个项目，禁止后台批量预取原始图片或所有实况视频。

**[FR-502] 缓存与续传读取**

- 受保护的不可变派生文件返回 `ETag` 或等价校验标识，并允许浏览器复用已验证缓存。
- 视频派生文件支持 Range 分段读取；无效 Range 返回明确状态，不退化为重复下载完整文件。
- 缩略图、预览图和实况派生文件使用独立 URL，避免缓存键互相覆盖。

**[FR-503] 加载与失败状态**

- 用户触发媒体查看或实况播放后 300 ms 内出现不遮挡主体的加载状态。
- 请求失败或超时后显示重试操作和简短原因；Live Photo 失败时保留完整尺寸的静态照片区域。
- 页面不得因为视频加载失败而阻塞左右切换、关闭查看器或下载原件。

### 5.7 个人文件夹整理

**[FR-601] 创建与管理**

- 每个账号拥有独立的个人文件夹列表，不向其他家庭成员暴露文件夹名称或分类关系。
- 文件夹名称为 1 到 40 个字符，同一账号下不允许仅大小写不同的同名文件夹。
- 用户可以删除自己创建的文件夹；删除只清除分类关系，不删除照片、视频、原件或派生文件。

**[FR-602] 批量归类**

- 用户进入明确的选择模式后，可以一次选择 1 到 100 个当前可见项目并加入指定文件夹。
- 同一媒体可以加入多个文件夹；重复加入同一文件夹时保持单条关系，不重复计数。
- 用户可以从当前文件夹批量移出项目，移出操作不改变媒体可见范围和磁盘文件。
- 账号只能归类家庭共享资产和自己拥有的私密资产，不能通过构造请求访问其他成员的私密资产。

**[FR-603] 文件夹浏览**

- 文件夹标签显示名称和项目总数，并支持在移动端横向滚动。
- 进入文件夹后仍可切换“家庭共享”和“仅自己可见”，两种范围分别展示文件夹内对应资产。
- 文件夹为空、被删除或加载失败时提供明确状态，不影响返回全部照片。

### 5.8 拍摄时间与时间轴

**[FR-701] 拍摄时间提取**

- 图片和 Live Photo 上传完成后，优先从静态原件读取 `DateTimeOriginal`、`CreateDate` 等标准元数据。
- 无有效拍摄时间时使用上传完成时间作为排序和月份归档依据，不阻止原件保存或预览生成。
- v0.5.0 启动时为既有图片和 Live Photo 补录可读取的拍摄时间，只更新 SQLite，不移动或改写原件。

**[FR-702] 时间轴查询**

- 资产按“拍摄时间优先、上传时间兜底”倒序排列，并使用同一字段完成游标分页。
- 后端提供当前共享/私密范围和个人文件夹下的月份及项目数量统计。
- 月份筛选与账号权限、共享/私密范围、文件夹筛选同时生效。
- 前端可使用月份统计生成桌面端真实时间比例 scrubber；年份按日历时间跨度分布，有照片的月份显示可点击小点，悬停小点只显示月份，不显示各月份照片数量。

**[FR-703] 响应式浏览**

- 桌面端提供常驻侧栏，移动端隐藏侧栏并使用适合触控的横向文件夹栏和月份选择器。
- 时间轴与全部照片使用独立布局；切换视图不得改变资产、文件夹或权限数据。
- 从时间轴打开查看器时，左右切换顺序必须与当前时间排序一致。

**[FR-704] 精选集入口**

- 登录后的默认首页为“精选”，不再把“照片墙”和“全部照片”作为重复入口。
- 精选集由服务端基于现有 `assets` 元数据派生，不新增物理目录，不复制原始文件，不改变文件夹、权限或最近删除状态。
- 精选页只提供回忆类入口，首期固定提供今日往年和本月回忆；最近导入、待整理、实况、视频、防窥属于全部照片筛选，收藏属于侧边栏入口。
- 精选页可使用类似 Apple 相册“精选/回忆”的轮转照片墙：左右照片列使用精选集封面，中间保留可点击的回忆入口；移动端降级为稳定的单列入口和双列照片墙。
- 精选集数量、封面和点击后的列表必须遵守当前账号、家庭共享/仅自己可见、最近删除和防窥边界。
- 今日往年和本月回忆只使用 `shooting_time` 计算；缺少拍摄时间的照片不进入这两个回忆入口，避免上传时间误导。
- 防窥照片可以出现在回忆类精选中，但封面层必须继续模糊或遮挡，不直接展示清晰缩略图。
- 点击精选集后进入普通列表，继续复用分页、密度切换、选择模式、标签、文件夹归类和查看器。

**[FR-705] 共用展示页**

- 顶部 `MintGallery` 文字或叶子图标作为展示页入口，进入后显示独立的沉浸式图片浏览界面。
- 展示页默认使用已收藏的家庭共享照片；如果管理员保存了展示配置，则所有用户看到同一套配置。
- 管理员可在展示页选择最多 13 张图片作为全局展示集；清空配置后回退到收藏默认图。
- 可选素材必须是家庭共享、非防窥、未删除、已处理完成的图片或 Live Photo 静态图；普通视频不进入首版展示页。
- 展示页不创建新的媒体副本，不移动原始文件，不改变收藏、文件夹、标签、最近删除或防窥语义。
- 点击展示图后复用现有媒体查看器，继续支持标签、信息面板、收藏和非破坏式图片编辑。

### 5.9 搜索、信息面板与防窥隐私

**[FR-801] 范围内搜索**

- 搜索入口位于相册主浏览界面，不进入单独页面。
- 搜索只在当前账号有权查看的资产中生效，并继续叠加“家庭共享/仅自己可见”、个人文件夹和月份筛选。
- v0.7.1 至少支持按标签、上传者和媒体类型（照片、视频、实况）搜索；原始文件名可以作为技术兜底搜索字段，但不作为主要用户入口。
- 搜索结果中的防窥照片仍按防窥样式显示，不因搜索命中而直接展示清晰缩略图。

**[FR-802] 照片信息面板**

- 查看器提供照片信息入口，优先展示用户设置的标签，并展示媒体类型、文件大小、拍摄时间、上传时间、上传者、可见范围和防窥状态。
- 原始上传文件名保留在数据库中用于追溯和下载，但信息面板不应把它作为主展示字段。
- 上传者或管理员可以在信息面板中修改标签、可见范围和防窥状态；普通成员不能修改他人上传的资产。
- 修改可见范围后必须立即刷新当前视图和月份统计，不能让已移出当前权限范围的照片继续留在列表中。

**[FR-803] 防窥隐私照片**

- 防窥隐私用于减少旁人瞥屏风险，不等同于访问权限，不改变媒体文件的后端可读权限。
- 外层浏览界面必须对防窥照片使用模糊或马赛克遮挡，并显示防窥标识。
- 点击防窥照片进入查看器后，仍必须先显示遮挡层和“查看原图”操作；只有用户明确点击后才显示清晰预览或允许 Live Photo 动态播放。
- 关闭查看器、切换到另一张照片或重新打开防窥照片时，遮挡状态必须恢复。
- 选择模式支持批量设为防窥、取消防窥、设为家庭共享和设为仅自己可见。

**[FR-804] 收藏、最近删除与批量标签**

- 已登录且有权查看某资产的成员可以收藏或取消收藏该资产；收藏作为家庭共同标记，不改变共享/私密可见范围。
- 收藏入口只显示当前账号有权查看且未进入最近删除的收藏资产。
- 上传者或管理员可以将资产移入最近删除；普通成员不能删除他人上传的资产。
- 最近删除是软删除：资产从普通图库、时间轴、文件夹、搜索结果和共享媒体访问中移出，但数据库记录、原始文件和派生文件保留。
- 上传者或管理员可以从最近删除恢复资产；恢复后资产回到原来的共享/私密范围和文件夹关系中。
- 选择模式支持批量覆盖所选资产标签；标签清洗、去重和数量限制继续由服务端统一执行。

## 6. 数据与存储设计

### 6.1 本地目录

生产数据根目录必须通过配置指向 Git 仓库之外的独立磁盘目录，内部继续使用相对路径组织文件：

```text
F:/MintGallery/data/
  originals/{userId}/{year}/{month}/{assetId}/
  derivatives/{assetId}/
  temporary/{uploadId}/
  database/
```

- 数据库只保存相对路径，不保存依赖某台电脑盘符的绝对路径。
- 启动脚本必须拒绝位于项目仓库内部的数据根目录；最终盘符由部署时显式确认。
- 原始文件、派生文件和临时文件分开存放。
- 共享或私密权限不通过物理目录直接暴露。
- 磁盘剩余空间不足时拒绝开始新上传，并保留明确错误信息。

### 6.2 核心数据表

**`users`**

| 字段 | 说明 |
| :--- | :--- |
| `id` | 用户 UUID |
| `username` | 唯一登录名 |
| `password_hash` | 密码哈希 |

| `favorite` | 是否被家庭共同收藏 |
| `tags` | 用户手动设置的展示标签，按规范化 JSON 数组存储 |

## 7. 安全要求

**[NFR-101] 媒体访问控制**

- 浏览器通过安全会话 Cookie 访问受保护媒体，或使用短时有效的签名 URL。
- 后端必须逐次检查资产可见范围，不能把媒体目录作为公开静态目录。
- 不向前端返回服务器真实磁盘路径。
- 视频接口支持鉴权后的 Range 请求。

**[NFR-102] 账号安全**

- 密码使用 Argon2id 或等价的现代密码哈希算法保存。
- 登录接口具备速率限制和失败日志。
- Cookie 会话启用 CSRF 防护；公网阶段必须启用 HTTPS。

**[NFR-103] 上传安全**

- 限制单文件大小、单次队列数量和允许的真实文件类型。
- 清理文件名并阻止目录穿越。
- 异常、磁盘写满或处理中断时，不生成半完成的正式资产记录。

**[NFR-104] GitHub 安全**

- `.gitignore` 必须排除媒体目录、SQLite 数据库、环境变量文件、日志和临时文件。
- 提交前检查密码、密钥、会话密钥和私人文件。
- GitHub 仓库优先设为私有，但私有仓库也不能用于提交家庭照片。

**[NFR-105] Tailscale 私有访问**

- Fastify 只监听 `127.0.0.1`，不直接开放局域网或公网端口。
- Tailscale Serve 负责 tailnet 内的 HTTPS 终止和访问路径，不使用 Funnel。
- 只有管理员批准加入同一 tailnet 的设备才能访问相册地址。
- 服务端 Cookie 在 HTTPS 模式启用 `Secure`，账号验证仍由 MintGallery 自身完成。
- Windows 电脑必须配置开机启动、登录触发启动、周期健康检查、运行日志和健康检查日志。
- 健康检查至少验证本机 `/api/health`、Windows 任务状态、Tailscale 后端状态和 Tailscale Serve 是否仍代理到 `127.0.0.1:3000`。
- 当 Node 服务退出、任务未运行、Tailscale Serve 配置漂移或 Tailscale 卡在 `NoState/starting` 时，系统应尽量自动恢复；同时提供一个短入口脚本用于日常手动启动和诊断。电脑关机、休眠、断网、Tailscale 未登录或访问设备未加入 tailnet 不属于网页可自动修复范围。

**[NFR-106] 远程媒体传输**

- 验收时记录访问设备、浏览器、网络类型、Tailscale 直连或中继状态及实际吞吐，避免把链路问题误判为页面问题。
- 首屏和查看器不得请求当前视图不需要的原始媒体。
- 派生文件生成失败不影响原件完整性，所有派生文件都允许从原件重新生成。

## 8. 数据安全与备份策略

### 8.1 MVP 现状

MVP 默认只有家用电脑硬盘上的一个副本。该方案可以验证上传和浏览功能，但不能作为长期可靠的照片归档方案。硬盘损坏、误删除、勒索软件或电脑丢失都可能造成永久数据丢失。

### 8.2 推荐的后续方案

- 第二副本必须位于另一块物理硬盘或云端，不能只是同一硬盘的另一个分区。
- 至少备份原始媒体文件、数据库和必要配置；缩略图可以重新生成。
- 备份任务记录时间、文件数量、校验结果和失败原因。
- 定期执行恢复测试，确认备份不是只有记录而无法恢复。
- 后续接入网盘时应通过独立存储适配层实现，避免修改相册和权限核心逻辑。

## 9. 性能与兼容性

### 9.1 MVP 性能目标

以下指标在家用局域网、首批 30 个媒体项目、使用缩略图的条件下验收：

- 精选页首屏主要内容在 2 秒内可见。
- 单张缩略图建议控制在 100 KB 以内。
- 列表采用分页和懒加载，媒体总量增加时不会一次载入全部数据。
- 图片处理并发可配置，默认不超过 2 个任务，避免家用电脑内存骤增。
- 100 GB 已有媒体可以分批导入，单批失败不影响已经完成的文件。

### 9.2 v0.3.2 远程访问性能目标

以下时延指标以 Tailscale 直连、有效吞吐不低于 10 Mbps、派生文件已生成的测试链路为基准；同时记录不依赖网速的请求行为指标：

- 全部照片首个可见缩略图在 3 秒内出现，且首屏不请求图片或视频原件。
- 打开单张照片后，屏幕预览在 2.5 秒内可见。
- 用户触发 Live Photo 后，动态播放在 3 秒内开始，不等待完整原始 MOV 下载完成。
- 单个 Live Photo 网页派生文件最长边不超过 1280 px，目标平均码率不高于 3 Mbps；实际规格以真实 iPhone 样本兼容测试为准。
- Tailscale 中继或低于基准吞吐时不承诺固定时延，但必须显示加载进度、允许重试，并始终保留静态照片浏览能力。

### 9.3 兼容性目标

- 优先支持近期版本的 iOS Safari、Android Chrome 和桌面 Chrome/Edge。
- HEIC、MOV 和 Live Photo 必须使用真实 iPhone 样本测试，不能只使用改扩展名的模拟文件。
- 浏览器后台上传和视频编码兼容性属于平台限制，界面必须提供明确反馈和恢复方式。

公网 4G 下的首屏性能、弱网策略和跨地区访问在公网部署阶段另行定义，MVP 不承诺固定 1.5 秒指标。

## 10. MVP 验收标准

满足以下条件才视为本地存储 MVP 完成：

1. 管理员能够创建和停用家庭成员账号。
2. 未登录用户无法查看资产列表或通过猜测 URL 获取媒体文件。
3. 用户能够上传至少 50 个混合图片和视频文件，并看到逐项进度与失败重试。
4. 大文件上传中断后能够从已确认分片继续，而不是从零开始。
5. 上传后的原始文件 SHA-256 与客户端选择的文件一致，HEIC 原文件未被转码覆盖。
6. JPEG、PNG、WebP 和受支持的 HEIC 样本能够生成缩略图和预览图。
7. 普通 MP4/MOV 可以保存；浏览器支持的编码可以播放，不支持的编码可以下载。
8. 用户可以手动上传并确认一组 Live Photo 配对文件，网页能够播放或回退为静态图。
9. 共享资产对所有有效成员可见，私密资产不能被其他普通成员访问。
10. 磁盘空间不足、格式错误和处理失败均有可理解的错误提示，不留下可见的损坏资产。
11. 未配置第二副本时，界面始终显示“仅 1 个副本”，且不提示用户安全删除手机原件。
12. Git 仓库中不包含媒体、数据库、密码、密钥和临时上传文件。

### 10.1 v0.4.0 性能补充验收

1. iOS Safari、Android Chrome 和桌面 Chrome/Edge 中至少完成两类非宿主设备实测，并记录 Tailscale 链路状态。
2. 图片墙和图片查看器通过网络日志确认不隐式下载原始图片或原始 MOV。
3. 真实 iPhone Live Photo 可以通过 H.264 MP4 派生文件播放；不兼容或转码失败时静态照片、重试和原件下载仍可使用。
4. 视频 Range、条件缓存、会话鉴权和私密资产权限测试通过。
5. 现有 Live Photo 可以补生成派生文件，无需重新上传或修改原始文件。

### 10.2 v0.4.0 文件夹补充验收

5. 文件夹筛选继续执行共享/私密范围、分页和媒体访问鉴权。

### 10.3 v0.5.0 时间轴补充验收

1. 带 `DateTimeOriginal` 的新上传图片按拍摄时间排序；无拍摄时间的文件按上传时间兜底。

### 10.4 v0.5.1 可靠访问补充验收

1. Windows 任务计划中同时存在生产服务任务和周期健康检查任务。

3. 健康检查任务至少每 3 分钟检查本机健康接口、任务状态、Tailscale 后端状态和 Tailscale Serve 代理状态。

### 10.5 v0.6.0 外部文件夹导入补充验收

## 11. 技术实现建议

以下是实现建议，不作为用户可见功能：

- 前端：Vue 3、TypeScript、Vite、Vue Router、Pinia。
- 后端：Node.js、TypeScript、Fastify 或等价成熟框架。
- 数据库：SQLite，启用 WAL 模式和迁移机制。
- 图片处理：Sharp，并在目标系统验证 HEIC/HEIF 解码依赖。
- 断点续传：tus 协议及成熟服务端实现。
- 媒体存储：通过统一存储接口封装本地文件系统，后续增加网盘或对象存储适配器。
- 后台任务：MVP 使用数据库记录的持久化任务队列，限制并发并支持服务重启后恢复。

## 13. 已知限制

- 纯网页无法保证直接从 iPhone 照片选择器取得完整 Live Photo 双文件。
- 浏览器在锁屏或后台停留过久时可能暂停上传，断点续传只能在页面恢复后继续。
- 不转码时，部分 MOV 视频可能无法在非苹果浏览器中播放。
- 本地只有一个副本时，不能把该系统视为安全备份，也不建议删除手机上的唯一原件。
- GitHub Pages 只能托管静态前端，不能承担本项目的 Node.js 后端和本地媒体存储。
- Tailscale 模式要求访问设备安装客户端并加入同一 tailnet；电脑关机、休眠或断网时相册不可访问。
- Tailscale 无法建立直连时可能经 DERP 中继，速度受家庭宽带上行、中继路径和访问设备网络共同限制。
- v0.5.1 及后续健康检查可修复本机服务未运行、Tailscale Serve 配置漂移和部分 Tailscale `NoState/starting` 卡住状态，但不能修复电脑休眠、关机、家庭宽带断网、Tailscale 未登录或访问设备未连接 tailnet。
- Live Photo 自动从 iPhone 相册提取仍暂缓；v0.6.0 只在外部文件夹扫描中识别同文件夹同名双文件候选。
- 个人文件夹是 SQLite 中的逻辑分类，不会在 `F:\MintGallery\data` 下创建同名磁盘目录，也不会移动原件。
- 缺少或无法解析拍摄时间的媒体会归入上传时间对应的月份；v0.5.0 尚不提供手动修改拍摄时间。
- 外部文件夹扫描是一次性导入，不是持续同步；源文件后续修改不会自动同步到相册。
- 防窥隐私只用于减少瞥屏风险，不是端到端加密或独立保密空间；真正访问权限仍由“家庭共享/仅自己可见”和后端鉴权决定。
- 最近删除是软删除，本版本不会从磁盘永久删除原始文件，也不能替代备份。
